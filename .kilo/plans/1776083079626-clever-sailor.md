# Plan: Model Selection Before Prompt Entry

## Goal
Add a pre-prompt model selection flow so users choose provider/model before entering task text, while preserving existing non-interactive `memora run <task>` behavior.

## Current State (from repo inspection)
- `memora run` currently requires positional `<task>` and does not prompt interactively (`src/cli/commands/run.ts`).
- Config already stores one `model` per provider, but runtime does not expose model selection UX (`src/config/schema.ts`).
- Provider adapters are stubs and currently ignore model configuration in execution (`src/providers/*.ts`).
- Orchestration/trace persist task and run details, but not explicit provider/model choice (`src/runtime/orchestrator.ts`, `src/trace/trace-writer.ts`).

## Scope and Decisions
- Implement pre-prompt selection for `memora run` when task text is omitted.
- Keep `memora run <task>` fully supported for scripts/automation.
- Add explicit CLI overrides (`--provider`, `--model`) for deterministic non-interactive usage.
- Persist selected provider/model into trace payload for explainability.
- Do not add DB migration in this iteration; keep selection metadata in trace + command string only.

## UX Specification
1. `memora run <task>`
   - No interactive prompt.
   - Provider/model resolved via CLI overrides or config defaults.
2. `memora run` (no task)
   - Step 1: Select provider.
   - Step 2: Select model for selected provider.
   - Step 3: Enter task prompt text.
   - Then execute existing run pipeline.
3. `memora run --provider <name> --model <name>`
   - If task is omitted, only ask for task prompt (provider/model already fixed).
4. Non-TTY behavior
   - If task omitted and stdin is not interactive, fail fast with a clear error (no hanging).

## Implementation Plan

### 1) Extend config model options (backward compatible)
- Update provider schemas to support optional `models: string[]` while retaining existing `model` as default.
- Validation rules:
  - `models` values are non-empty strings.
  - If `models` exists, configured default `model` must be present in `models`.
  - Deduplicate/normalize only via validation error (no silent mutation).
- Files:
  - `src/config/schema.ts`
  - `tests/config/schema.test.ts`

### 2) Add run-time selection resolver
- Add helper that resolves `{ provider, model }` from:
  1) CLI overrides,
  2) config default provider + default model,
  3) optional interactive answers.
- Validate provider via existing allowlist (`assertSupportedProvider`).
- Validate model against provider model set:
  - Allowed set = `providers[provider].models` if present, else `[providers[provider].model]`.
- Place helper in run command module or a small utility module under `src/cli/`.

### 3) Implement interactive pre-prompt flow in `run` command
- Change run command argument from required to optional (`[task]`).
- Add options:
  - `--provider <provider>`
  - `--model <model>`
- Use `node:readline/promises` to prompt when needed.
- Prompt order must always be provider/model first, prompt text second.
- Keep existing init + orchestration path unchanged once resolved.
- Files:
  - `src/cli/commands/run.ts`

### 4) Propagate selection through runtime contract
- Extend `RunTaskOptions` with `provider` and `model` fields.
- Attach selected provider/model to early trace events (`run.started` payload and/or `CONTEXT_READY` note).
- Include provider/model in command metadata string used for run persistence for auditability.
- Files:
  - `src/runtime/orchestrator.ts`
  - `src/trace/trace-writer.ts` (schema if new top-level trace fields are added)
  - `src/trace/explain-service.ts` (optional display of selected model/provider)

### 5) Keep provider abstraction future-ready
- No full provider runtime integration needed now; model selection is UX and metadata wiring.
- Optional light follow-up: add `model?: string` to `ProviderCompleteRequest` for future usage without immediate behavior changes.
- Files (optional in this slice):
  - `src/providers/base.ts`
  - `tests/providers/registry.test.ts`

## Testing Plan

### Unit tests
- `tests/config/schema.test.ts`
  - accepts valid `models` arrays,
  - rejects default `model` not included in `models`,
  - still accepts legacy config without `models`.

### CLI integration tests
- `tests/cli/run-command.test.ts`
  - existing non-interactive `run <task>` tests remain green.
  - add test for `run --provider ... --model ... <task>` success path.
  - add test for `run` without task in non-TTY mode -> clear failure message.
  - add interactive test (`spawn` + stdin writes) verifying order: provider/model selected before task entry.

### Trace/explain tests
- `tests/trace/explain-service.test.ts`
  - verify selected provider/model appears in explain output when present.

## Acceptance Criteria
- Users can choose provider/model before entering prompt in interactive `memora run` flow.
- Existing scripted usage (`memora run <task>`) remains backward compatible.
- Invalid provider/model combinations fail with actionable errors.
- Selected provider/model is captured in trace for post-run explainability.
- Test suite updated and passing for new behavior.

## Risks and Mitigations
- Interactive test flakiness: use deterministic stdin script and explicit prompt markers.
- Config compatibility risk: keep `model` required and `models` optional.
- Future mismatch with provider SDK integration: keep resolver centralized to reuse later when adapters become real.

## Suggested Execution Order
1. Config schema/tests.
2. Run command resolver + CLI options.
3. Interactive prompting flow.
4. Runtime/trace metadata wiring.
5. Integration + trace tests.
6. Final regression (`npm run build`, `npm test`).
